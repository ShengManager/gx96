# Telegram Bot Message ID 与重复发送流程分析

## 结论先说

当前 `messageId` 出现 `490 -> 492`（跳过 `491`）本身**不一定是 bug**，但在你这个案例里，结合运行日志可以判断：

1. **同一个 Bot 有多实例在同时处理更新**（关键证据：持续 `409 conflict`）。
2. 多实例并发时，会发生“两个实例都处理同一个功能点击”，导致：
   - 先后发送多条消息（或一条被马上删掉）；
   - 数据库里 `messageId` 出现跳号；
   - 聊天窗口偶发看到双消息（如 Bonus 列表重复）。

换句话说：`490 -> 492` 是“并发发送/删改竞争”的可见现象，不是根因。根因是**多实例并发消费同一 token 更新**。

---

## 你问的重点：为什么不是 491？

在 Telegram 私聊里，`message_id` 是按聊天递增的，但**任何消息事件都会占用 ID**。  
所以当你看到 `490 -> 492`，通常是以下两种情况之一：

- `491` 被另一条消息占用（可能是并发实例发送的消息）；
- `491` 被发送后又被快速删除（你肉眼只看到 `492`）。

在你当前项目中，更符合第二种和并发竞争场景：

- 实例 A / B 同时收到或处理相近时序的 callback；
- A 发送了 `491`；
- B 又发了 `492`，并把 A 产生的消息当作“旧消息”删掉，或两边互相覆盖；
- 最终界面可能看到重复，也可能只剩后一条，但 ID 已跳。

## 目前代码的实际处理流程（`server/services/telegramBot.ts`）

### 1) Callback 入口

`bot.on("callback_query", ...)` 中做了：

- callback 去重（最近事件缓存）
- chat 级节流（500ms）
- action 级节流（500ms）
- in-flight 锁（chat/action）

然后开始处理业务分支（`main_menu`、`games`、`bonus` 等）。

### 2) 发送前删除旧消息

目前发送会走统一的持久化逻辑（`ACTIVE_VIEW_KEY = "__active__"`）：

- 先读 `telegram_chat_views` 里该 chat 的上次 `messageId`
- 尝试删除旧消息
- 发送新消息
- 回写新的 `messageId`

### 3) Main Menu 特殊处理

`showMainMenu(...)` 里还有额外的串行队列和删除逻辑，防止主菜单并发渲染。

## 关键证据（你环境中已出现）

从运行终端可见：

- 持续出现 `Polling conflict (409) - another instance may be running`

这意味着至少存在“另一个 bot polling 实例”在抢同一 token。  
在这种情况下，单进程内的锁（Map/Set）无法跨进程生效，重复处理会继续发生。

---

## 为什么你数据库里还看到 `main_menu/games/bonus` 的 `viewKey`

当前代码已经改为统一使用 `__active__`。  
如果数据库仍持续出现 `main_menu/games/bonus` 新记录，通常说明：

- 旧版本进程还在跑（仍按旧逻辑写入）；
- 或有第二个实例不是最新代码。

这进一步印证“多实例并发”问题。

---

## 建议的排查顺序（先做这个，再谈代码细修）

1. **确保只保留一个 bot 进程**
   - 本机只跑一个 `pnpm dev`。
   - 关闭其他终端、PM2、Docker、旧服务、远端副本（若也在跑同 token）。
2. **观察日志必须不再出现 409**
   - 只要还有 409，就仍有多实例冲突。
3. **重启后再测**
   - `/start -> Games -> Back -> Bonus -> Back` 连续点击测试。
4. **检查 `telegram_chat_views`**
   - 期望单 chat 只保留 `__active__` 记录（旧键会逐步被清理）。

---

## 你这个问题的本质

你现在遇到的不是“单纯代码一处 if/else 出错”，而是**分布式并发问题（多实例消费同一 Telegram 更新）**。  
只要多实例不收敛到单实例，`messageId` 跳号与偶发双发就会反复出现。

但第一优先级仍是：**先解决 409 多实例冲突**。
